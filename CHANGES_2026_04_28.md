# Stratos1 — Strict architecture refactor (2026-04-27 → 2026-04-28)

Plain-English explanation of what was deleted, what was added, and why.

The driving rule, given by the client on 2026-04-28:

> The bot must not think, assume, guess, or infer anything.
> Everything must be controlled by Bybit and verified through real Bybit events.
> No fallback logic, no parallel systems, no dead code.

---

## What was DELETED (and why)

**Every place the bot was making its own decisions about closes or SL moves.**

1. **`max_loss_cap` (-4 USDT bot-side close)**
   The bot watched unrealized PnL on every tick and force-closed any trade whose PnL crossed -4 USDT. This was the single biggest source of the bleeding on 2026-04-27 — 4 of the 5 worst-loss trades hit it. Bybit decides closes now, not the bot.

2. **`combined_loss_cap` (-4 USDT main+hedge bot-side close)**
   Same shape: the bot summed main + hedge PnL and force-closed when the combined loss got too negative. Removed for the same reason.

3. **`BreakevenManager`**
   Polled price every tick. When the trade went +2.3% in profit, the bot moved the SL to break-even. The decision was bot-side; only the SL update happened on Bybit. Removed entirely.

4. **`TrailingManager`**
   Same shape as the breakeven manager — bot polled price, when the trade reached the activation level the bot called `set_trading_stop` to start the trailing. Removed.

5. **`_check_tp_progression`** (TP-2 SL advancement)
   Bot watched price, detected when each TP was hit, then moved the SL to a previous TP. Pure bot inference. Removed. The partial TP closes themselves are still placed on Bybit as conditional orders — Bybit fires those autonomously.

6. **`_infer_close_reason`** (close-reason guesser)
   When Bybit closed a position, this function compared the exit price against SL / BE / TP / avg_entry and *guessed* what closed it. That's where the misleading `near_entry_close` and `tp_progression_sl_hit` labels came from. Removed.

7. **30-second position polling** in the trigger loop
   The bot ran a `get_position` REST call every 30 seconds for each open trade as a fallback to the WebSocket position update. Per the no-fallback rule, removed.

8. **15-second hedge polling** (`HedgeManager.check_linked_closure`)
   Same shape — fallback poll. Removed.

9. **`HedgeManager.check_pending_sl_move`** (post-hedge SL move polling)
   Bot polled price after a hedge opened, decided when to nudge the original SL closer to entry. Removed.

10. **`ReentryManager.check_and_activate`** (re-entry price polling)
    Bot watched price and decided when to re-enter after a stop-out. Removed.

11. **Dead notifier templates**
    `stop_loss_hit`, `break_even_adjusted`, `sl_moved`, `hedge_denied` — all only fired from code that was deleted above. Cleaned out so nothing in the codebase pretends to do something it doesn't.

12. **Dead config**
    `WalletSettings.max_loss_usdt`, `HedgeSettings.max_combined_loss_usdt`, `BreakevenSettings`, `SafetyLadderStep`, the `[breakeven]` block in `config.toml`, BE rows in `SETUP.md`, BE startup print in `test_live.py`. None of it was being read anymore.

## What was ADDED (and why)

1. **Bybit-native trailing stop, set ONCE at trade open**
   When the trade opens, the bot calls Bybit `set_trading_stop` with a trailing distance and an activation price. From that point Bybit owns the trade's upside management — no bot polling.

   Activation rule: `min(highest TP, +6.1%)` for LONG / `max(lowest TP, -6.1%)` for SHORT (client spec 2026-04-28: *highest TP first, 6.1% second*). All TPs at or above 6.1% from entry merge into the trailing; TPs below 6.1% get individual partial-close conditional orders.

2. **Close-reason classification driven entirely by Bybit's order-fill event**
   When `on_order_update` reports `status=Filled` the bot reads `stopOrderType` / `execType` on the fill. The mapping is mechanical:

   | Bybit fill event | Internal reason | Telegram header |
   |---|---|---|
   | `stopOrderType=StopLoss` | `stop_loss` | `POSITION CLOSED - stop loss` |
   | `stopOrderType=TrailingStop` | `trailing_stop` | `POSITION CLOSED - trailing stop` |
   | `stopOrderType=Stop` + matches a recorded TP order | `tp_N` (N = TP level) | `POSITION CLOSED - TPN` |
   | `execType=Liquidation` | `liquidation` | `POSITION CLOSED - liquidation` |
   | nothing classified | `external_close` | `POSITION CLOSED - external close` |

   The bot never invents the reason — it comes from Bybit's fill event.

3. **Per-TP "TAKE PROFIT N TAGEN" notification driven by Bybit's order-fill event**
   When Bybit reports a Filled event for one of the partial-TP conditional orders the bot placed, the bot looks up which TP level by matching the `triggerPrice` to `signal.tps`, and emits the Swedish notification. No bot-side TP detection.

4. **Re-entry triggered by Bybit's SL-fill event**
   `close_trade` checks if the reason is `stop_loss`. If yes, and re-entries aren't exhausted, it calls `ReentryManager.activate_after_sl(trade)`, which opens a fresh trade at market through `process_signal`. No polling.

5. **Single `position_closed` Telegram template** with `close_source` parameter appended to the header. Replaces the previous split between `stop_loss_hit` and `position_closed`.

6. **Leftover-order sweep on close**
   When `close_trade` runs, the bot also cancels every open conditional/limit order sitting on the closing position's `(symbol, positionIdx)`. Bybit doesn't auto-cancel untriggered Stop orders when the position is gone, so without this sweep they accumulate (the 29 orphan orders observed on 2026-04-28).

7. **Backtick stripping in the parser**
   AI Felix Crypto and a few other channels wrap prices in backticks (`` `449.33` ``). The entry/SL/TP regex patterns expected bare numbers, so the backticks broke extraction. One-line fix mirroring the `**` strip from yesterday.

8. **`scripts/clean_state.py`** — one-command cleanup after every update
   Closes all Bybit positions, cancels all open orders, marks DB ghost trades CLOSED, deletes WishingBell messages. Run it after each deploy so the operator can verify fresh behaviour from a known clean baseline.

---

## What stayed the same

* Signal pipeline: parse → duplicate check → leverage calc → market entry → set Bybit SL + partial TPs + arm trailing.
* Hedge pre-arm: still placed as a Bybit conditional order at trade open.
* Re-entry counts and exhaustion notification: same Swedish templates, same limits.
* All the duplicate-detector logic, signal parser, leverage calc, and signal-saved DB plumbing.

---

## How to verify the bot is "100% clean" after an update

```bash
ssh -i <key> root@vps
cd /opt/stratos1
git pull origin main
systemctl restart stratos1
venv/bin/python scripts/clean_state.py
```

The `clean_state.py` output should show `closed=0 cancelled=0 ghost=0 deleted=0`
the second time it's run — that's the canonical "clean".
